# Оформление КД для серийного производства

Конструкторская документация для единичного заказа и для серийного производства — это два принципиально разных уровня проработки. Единичный светильник можно собрать, держа в голове нюансы и уточняя детали у конструктора лично. Серия из 500 штук требует документации, по которой любой сборщик на производстве соберёт изделие без единого вопроса. В этом уроке — как перейти от документации "для себя" к документации "для производства".

---

## Почему единичная КД не работает на серии

При единичном производстве конструктор часто выступает и автором документации, и консультантом на сборке. Если что-то непонятно — сборщик подходит и спрашивает. Такой подход ломается при масштабировании.

Типичные проблемы, когда серию запускают по "единичной" документации:

* Сборщик не понимает последовательность операций и собирает в произвольном порядке — получает брак
* Нет указаний по контролю качества на промежуточных этапах — дефект обнаруживается только на финальной проверке
* Конструктор становится "бутылочным горлышком" — производство стоит, пока он не ответит на вопросы
* При замене сборщика новый человек не может разобраться в документации без обучения

> Правило: если для сборки изделия нужно присутствие конструктора — документация не готова к серии.

---

## Комплект документов для серийного производства

Для единичного заказа обычно достаточно сборочного чертежа и спецификации. Серийное производство требует расширенного комплекта.

| Документ | Единичное | Серийное | Назначение |
|----------|-----------|----------|------------|
| Сборочный чертёж | Да | Да | Общий вид, габариты, посадки |
| Спецификация | Да | Да | Перечень деталей и покупных |
| Чертежи деталей | Частично | Все | Изготовление каждой детали |
| Маршрутная карта | Нет | Да | Последовательность операций |
| Операционная карта | Нет | Да (для сложных операций) | Детальное описание операции |
| Ведомость комплектации | Нет | Да | Что положить на рабочее место |
| Карта контроля | Нет | Да | Что и как проверять |

## Маршрутная карта

**Маршрутная карта** (МК) — главный документ для организации серийной сборки. Она описывает весь путь изделия от первой операции до упаковки.

### Что содержит маршрутная карта

* Номер и название операции
* Рабочее место или участок
* Перечень деталей и материалов на операцию
* Инструмент и оснастка
* Норма времени (сколько минут на операцию)
* Точки контроля качества

### Пример маршрутной карты светильника

Рассмотрим типовой светильник 7Lamp линейного типа — алюминиевый профиль, светодиодная линейка, драйвер, рассеиватель.

| N | Операция | Участок | Время, мин | Контроль |
|---|----------|---------|------------|----------|
| 005 | Входной контроль комплектующих | Склад | 10 | Визуальный осмотр, проверка партии |
| 010 | Нарезка профиля в размер | Мехобработка | 5 | Длина +/- 0.5 мм |
| 015 | Установка торцевых заглушек | Сборка | 3 | Плотность посадки |
| 020 | Монтаж LED-линейки на теплопроводящий скотч | Сборка | 5 | Равномерность прижима |
| 025 | Пайка / подключение проводов | Электромонтаж | 8 | Качество пайки, полярность |
| 030 | Подключение драйвера | Электромонтаж | 5 | Схема подключения |
| 035 | Первичное включение и проверка | Тестирование | 3 | Ток, напряжение, равномерность |
| 040 | Установка рассеивателя | Сборка | 2 | Фиксация без зазоров |
| 045 | Финальная проверка | ОТК | 5 | Визуал, электробезопасность |
| 050 | Маркировка и упаковка | Упаковка | 5 | Комплектность, этикетка |

Нумерация операций идёт с шагом 5 (005, 010, 015). Это стандартная практика по ГОСТ 3.1118 — она оставляет место для вставки дополнительных операций без перенумерации всей карты.

> На практике 7Lamp: маршрутные карты ведутся в ERP-системе. Каждая операция привязана к конкретному рабочему месту и исполнителю. Это позволяет отслеживать, на каком этапе находится каждое изделие в партии.

## Операционная карта

**Операционная карта** (ОК) — это подробная инструкция для одной конкретной операции из маршрутной карты. Её составляют для сложных или критичных операций, где недостаточно краткого описания.

### Когда нужна операционная карта

* Операция требует специфического инструмента или оснастки
* Есть строгая последовательность действий внутри операции
* Высокий риск брака при неправильном выполнении
* Операцию выполняют разные сборщики (нужна воспроизводимость)

### Структура операционной карты

1. Номер операции (совпадает с маршрутной картой)
2. Название операции
3. Пошаговые переходы (действия внутри операции)
4. Эскиз или фотография с указаниями
5. Инструмент для каждого перехода
6. Контрольные параметры

**Пример.** Операция 025 "Пайка / подключение проводов":

* Переход 1: Зачистить провод ШВВП 2x0.75 на 5 мм
* Переход 2: Облудить концы (температура паяльника 320 C)
* Переход 3: Припаять к контактным площадкам LED-линейки (красный — "+", синий — "-")
* Переход 4: Надеть термоусадку, усадить феном
* Переход 5: Проверить надёжность соединения (лёгкое усилие на отрыв)

## Ведомость комплектации

**Ведомость комплектации** отвечает на вопрос: что именно должно лежать на рабочем месте сборщика перед началом работы. Это не спецификация (перечень всех деталей изделия), а список для конкретного рабочего места на конкретную операцию или группу операций.

### Отличие от спецификации

| Спецификация | Ведомость комплектации |
|-------------|----------------------|
| Полный перечень деталей изделия | Набор для конкретной операции |
| Привязана к изделию | Привязана к рабочему месту |
| Содержит обозначения по ЕСКД | Содержит складские артикулы |
| Используется конструктором | Используется кладовщиком и сборщиком |

Ведомость комплектации упрощает работу склада: кладовщик собирает "набор" для каждого рабочего места, и сборщик не тратит время на поиск деталей.

## Нумерация документов

Без единой системы нумерации документация быстро превращается в хаос. Каждый документ должен иметь уникальный идентификатор, по которому его можно найти и однозначно соотнести с изделием.

### Принцип нумерации в 7Lamp

Структура обозначения: `7L.XXX.YYY.ZZZ`

* `7L` — префикс компании
* `XXX` — код серии или проекта (например, `LIN` для линейных, `DWN` для даунлайтов)
* `YYY` — номер изделия внутри серии (001, 002, ...)
* `ZZZ` — код документа

Коды документов:

| Код | Документ |
|-----|----------|
| СБ | Сборочный чертёж |
| СП | Спецификация |
| МК | Маршрутная карта |
| ОК | Операционная карта |
| ВК | Ведомость комплектации |
| ПС | Паспорт изделия |

Пример: `7L.LIN.003.МК` — маршрутная карта для третьего изделия линейной серии.

> Нумерация привязана к ERP-системе. При создании нового изделия система автоматически присваивает следующий номер в серии и генерирует шаблоны документов.

## Ревизии документов

Серийное производство живёт долго. За время выпуска серии неизбежно появляются изменения: замена комплектующих, оптимизация сборки, исправление ошибок. Каждое изменение должно быть отслеживаемым.

### Правила ведения ревизий

* Первая выпущенная версия документа — ревизия **A**
* Каждое последующее изменение увеличивает букву: **B**, **C**, **D**
* В штампе чертежа всегда указана текущая ревизия
* Предыдущие ревизии хранятся в архиве (не удаляются)
* На производстве всегда должна быть только актуальная ревизия

### Что фиксируется при смене ревизии

* Дата изменения
* Автор изменения
* Краткое описание: что изменилось и почему
* Список затронутых документов (одно изменение может затронуть чертёж, маршрутную карту и спецификацию одновременно)

## Извещения об изменениях

**Извещение об изменении** (ИИ) — формальный документ, который запускает процесс внесения правок в действующую КД. Без извещения никто не имеет права менять документацию, по которой уже идёт серийное производство.

### Процесс внесения изменений

1. Конструктор выявляет необходимость изменения (или получает запрос от производства)
2. Составляет извещение об изменении с описанием: что, где, почему
3. Извещение согласовывается (руководитель проекта, технолог, при необходимости — отдел закупок)
4. После согласования конструктор вносит правки в документацию и повышает ревизию
5. Обновлённые документы передаются на производство, старые изымаются
6. В ERP фиксируется связь: извещение — документ — ревизия

### Типичные причины изменений

* Замена комплектующих (поставщик снял с производства)
* Оптимизация технологии сборки (сокращение времени операции)
* Устранение выявленного дефекта
* Требования заказчика (изменение характеристик)
* Снижение себестоимости (замена материала на более дешёвый без потери качества)

> Частая ошибка: конструктор правит чертёж "по-тихому", не выпуская извещение. На производстве продолжают работать по старой версии, потому что не знают об изменении. Результат — партия брака.

## Чеклист: готовность КД к серии

Перед запуском серийного производства проверьте комплектность документации:

- [ ] Сборочный чертёж с актуальной ревизией
- [ ] Чертежи всех деталей собственного изготовления
- [ ] Спецификация с указанием покупных и собственных деталей
- [ ] Маршрутная карта с полной последовательностью операций
- [ ] Операционные карты для сложных и критичных операций
- [ ] Ведомость комплектации для каждого рабочего места
- [ ] Карта контроля качества с параметрами и допусками
- [ ] Нумерация всех документов по единой системе
- [ ] Документация проверена человеком, не участвовавшим в разработке

Последний пункт — критически важный. Лучший тест для документации: дайте её сборщику, который никогда не видел это изделие. Если он соберёт светильник без вопросов — КД готова.

## Практика 7Lamp: как устроен процесс

В 7Lamp переход от единичного заказа к серии проходит в три этапа.

**Этап 1. Опытный образец.** Конструктор собирает первый экземпляр лично. Фиксирует реальную последовательность сборки, замеряет время операций, отмечает "узкие места".

**Этап 2. Пробная партия (3-5 штук).** Сборку выполняет производственный персонал по черновой маршрутной карте. Конструктор присутствует, но не вмешивается — только записывает вопросы сборщиков. После пробной партии маршрутная карта корректируется.

**Этап 3. Запуск серии.** Полный комплект КД передаётся на производство. Конструктор больше не участвует в ежедневной сборке. Все изменения — только через извещения.

Такой подход позволяет "отладить" документацию до запуска полноценной серии, а не исправлять её по ходу, теряя время и материалы.
